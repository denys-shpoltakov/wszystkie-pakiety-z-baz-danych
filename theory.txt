====================================================================

1. Создание трех уровней модели: концептуального, логического и физического
Рекомендуется подготовить отдельные диаграммы: концептуальную — определяющую общие сущности и бизнес-отношения; логическую — более подробную, независимую от технологий; и физическую — адаптированную к конкретной системе баз данных. Такой подход экономит время во время изменений и облегчает понимание модели другими членами команды.

2. Применение последовательных конвенций именования
Названия таблиц, столбцов, ключей и т. д. должны соответствовать установленному шаблону. Это облегчает работу, а изменение имен в работающей базе данных может привести к ошибкам в приложениях или нарушить целостность данных.

3. Поддержание актуальной документации
Диаграммы и документация (например, словарь данных) должны постоянно обновляться, чтобы отражать изменения структуры и облегчать ее понимание.

4. Нормализация данных — минимизация избыточности
Проектирование должно ограничивать повторение информации и обеспечивать согласованность данных. Хранение повторяющейся информации в отдельных таблицах поддерживает целостность и улучшает производительность.

5. Выбор подходящих типов данных и ограничение количества полей
Выбор оптимальных типов данных и разумное количество столбцов (рекомендации говорят, например, о максимум ок. 150 полях в таблице) влияют на производительность и читаемость.

6. Определение первичных и внешних ключей
Каждая таблица должна иметь первичный ключ, определяющий уникальную запись, а отношения между таблицами — внешние ключи. Это поддерживает согласованность и обеспечивает логическую связь данных.

7. Индексы, ограничения и правила валидации
Хорошие практики включают использование индексов для более эффективного доступа, ограничений (constraints, например, CHECK, UNIQUE), а также валидацию на уровне таблицы, что улучшает качество данных.

8. Документирование изменений и версионирование схемы
Проект должен учитывать отслеживание изменений структуры и эволюцию схемы — изменения должны вноситься контролируемо, с отслеживанием версий.

9. Аудит данных (audit trail)
Добавление механизмов для отслеживания изменений в данных (например, поля с временем модификации) увеличивает прозрачность и позволяет отслеживать историю записей.

10. Модульность и повторное использование компонентов
Создавайте модульные элементы в БД: хранимые процедуры, представления, триггеры — что позволяет многократно использовать логику и облегчает обслуживание.

11. Простота и сотрудничество между командами
Ставьте на простоту модели (ее легче использовать пользователям), а также тесно сотрудничайте с администраторами БД (DBA) и программистами — как при проектировании, так и при последующих изменениях.

12. Понимание данных и паттернов использования
Перед проектированием необходимо понять: данные, их связи, способ использования и масштаб. Это предотвращает ошибки и обеспечивает оптимальный проект.

13. Противодействие ловушкам моделирования — 'fan trap' и 'chasm trap'
ERD (диаграмма сущность-связь) может вводить в заблуждение, если отношения построены неправильно. Fan trap (ловушка веера) заключается в неожиданном подсчете связей, а chasm trap (ловушка пропасти) — в пропущенных связях между сущностями.

14. Принцип ортогонального проектирования (POOD)
Отношения следует определять так, чтобы не представлять одни и те же факты разными способами — это устраняет избыточность и неоднозначность.

15. Рефакторинг схемы базы данных
Изменения в схеме (рефакторинг) следует вводить так, чтобы не менять семантику данных, т. е. система должна продолжать работать корректно на каждой фазе изменений.

16. Моделирование Data Vault (для хранилищ данных)
В случае хранилищ данных применяется модель Data Vault — она позволяет хранить все данные (в том числе несогласованные), с аудитом и отслеживанием источников, что улучшает масштабируемость и гибкость.

ХОРОШИЕ ПРАКТИКИ ОТНОСИТЕЛЬНО ИМЕНОВАНИЯ ТАБЛИЦ / СТРОК / SQL-ЗАПРОСОВ

1. Регистр букв в SQL
Ключевые слова SQL (например, SELECT, FROM, WHERE, JOIN, INSERT, UPDATE) - ЗАГЛАВНЫЕ БУКВЫ.
Благодаря этому они читабельны и их легко отличить от названий таблиц и столбцов.

Названия таблиц и столбцов - строчные буквы.
Последовательное использование строчных букв предотвращает проблемы в системах, чувствительных к регистру (например, Linux + MySQL).

Пример хорошего запроса: 
SELECT first_name, last_name
FROM users
WHERE status = 'active';

Пример ошибочного стиля:
select FirstName, LastName
From Users
where Status = 'active';


2. Множественное или единственное число?
Таблицы - множественное число (так как содержат много записей, например, users, orders, products).
Столбцы - единственное число (так как описывают одиночный атрибут записи, например, user_id, price, created_at).

Пример правильного именования (согласно конвенции):
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    first_name VARCHAR(50),
    last_name VARCHAR(50),
    email VARCHAR(100) UNIQUE
);

Пример неправильного запроса:
CREATE TABLE user (  -- единственное число, не подходит для набора данных
    users_id INT,    -- множественное число в колонке
    names VARCHAR(50)
);


3. Именование ключей и столбцов
Первичные ключи (Primary Key)
Имя: id или название_таблицы_id
Если таблица основная (например, users), можно использовать просто id.
В промежуточных таблицах (например, order_items) лучше использовать полное имя: order_item_id.

Внешние ключи (Foreign Key)
Имя: название_таблицы_id
например, в таблице orders - user_id (связь с users.id).

Пример:
CREATE TABLE orders (
    order_id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    total DECIMAL(10,2),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);


4. Названия столбцов
- Читаемые, описательные, строчные буквы.
- Snake_case (подчеркивания) вместо camelCase (first_name, а не firstName).
- Не используйте национальные символы (например, польские или русские буквы) или пробелы (imie вместо imię).
- Не используйте зарезервированные слова (например, order, select, group). Если необходимо, используйте обратные кавычки: `order`.
- Даты и время -> с суффиксами _at (created_at, updated_at).
- Флаги/логические -> префикс is_ или has_ (is_active, has_discount).
- Суммы, числовые значения -> описательные единицы (price, quantity, amount).

5. Именование индексов, констрейнтов и отношений
Primary Key -> pk_nazwatabeli
Foreign Key -> fk_tabela_kolumna
Unique Key -> uk_tabela_kolumna
Индекс -> idx_tabela_kolumna

Пример:
CREATE TABLE products (
    product_id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price DECIMAL(10,2) NOT NULL
);

CREATE TABLE order_items (
    order_item_id INT AUTO_INCREMENT PRIMARY KEY,
    order_id INT NOT NULL,
    product_id INT NOT NULL,
    quantity INT DEFAULT 1,
    FOREIGN KEY fk_order_items_order (order_id) REFERENCES orders(order_id),
    FOREIGN KEY fk_order_items_product (product_id) REFERENCES products(product_id)
);

6. Стиль запросов
- Отступы и форматирование: каждая клаузула (SELECT, FROM, WHERE, JOIN) в новой строке.
- Алиасы таблиц: короткие, но читаемые (u для users, o для orders).
- JOIN-ы: явно (INNER JOIN, LEFT JOIN), а не через запятые.

Пример хорошего стиля запроса:
SELECT o.order_id, u.first_name, u.last_name, p.name, oi.quantity
FROM orders o
INNER JOIN users u ON o.user_id = u.id
INNER JOIN order_items oi ON o.order_id = oi.order_id
INNER JOIN products p ON oi.product_id = p.product_id
WHERE u.is_active = 1
ORDER BY o.created_at DESC;

Пример менее читаемого запроса:
SELECT orders.order_id, users.firstName, products.name
FROM orders, users, order_items, products
WHERE orders.userId = users.id
AND order_items.orderId = orders.id
AND order_items.productId = products.id;

7. Чек-лист хороших практик:
- Ключевые слова SQL - ЗАГЛАВНЫЕ БУКВЫ
- Названия таблиц и столбцов - строчные буквы, snake_case
- Таблицы во множественном числе, колонки в единственном
- Primary key - id или название_таблицы_id
- Foreign key - название_таблицы_id
- Индексы и констрейнты - pk_, fk_, uk_, idx_
- Даты и время - _at (например, created_at)
- Логические флаги - is_, has_
- Читаемые отступы и алиасы в запросах

DML (Data Manipulation Language) - манипуляция данными
SELECT – выборка данных из таблицы.
INSERT – добавление новых строк.
UPDATE – модификация существующих строк.
DELETE – удаление строк.
REPLACE – работает как INSERT, но перезаписывает строку, если первичный/уникальный ключ уже существует.

DDL (Data Definition Language) - определение данных
CREATE DATABASE – создает базу данных.
DROP DATABASE – удаляет базу данных.
CREATE TABLE – создает таблицу.
ALTER TABLE – изменяет структуру таблицы (добавляет колонки, ключи).
DROP TABLE – удаляет таблицу.
TRUNCATE TABLE – удаляет все данные из таблицы (быстрее, чем DELETE).
CREATE INDEX – создает индекс.
DROP INDEX – удаляет индекс.

DCL (Data Control Language) - контроль доступа
GRANT – предоставление прав пользователю.
REVOKE – отзыв прав.
CREATE USER – создает пользователя.
DROP USER – удаляет пользователя.
SET PASSWORD – устанавливает пароль.

TCL (Transaction Control Language) - управление транзакциями
START TRANSACTION – начинает транзакцию.
COMMIT – подтверждает транзакцию.
ROLLBACK – откатывает транзакцию.
SAVEPOINT – точка сохранения внутри транзакции.
RELEASE SAVEPOINT – удаляет точку сохранения.

Операторы и условия:
WHERE – условия фильтрации.
AND, OR, NOT – логические операторы.
BETWEEN – проверяет диапазон значений.
LIKE – поиск по шаблону (например, %текст%).
IN – проверяет список значений.
IS NULL – проверяет отсутствие значения.
DISTINCT – возвращает уникальные строки.
ORDER BY – сортировка.
GROUP BY – группировка результатов.
HAVING – условия для групп (после GROUP BY).
LIMIT – ограничение количества результатов.
OFFSET – смещение результатов (используется, например, для пагинации).

----------------------------------------------------------------

JOIN – объединение таблиц.
INNER JOIN – только общие записи.
LEFT JOIN – все из левой таблицы + соответствующие из правой.
RIGHT JOIN – все из правой таблицы + соответствующие из левой.

----------------------------------------------------------------

Агрегатные функции:
COUNT() – количество строк.
SUM() – сумма значений.
AVG() – среднее значение.
MIN() – минимум.
MAX() – максимум.

Типы данных:

1. Числовые типы
INT – целые числа (например, 1, 25, -7).
BIGINT – очень большие целые числа.
DECIMAL(m,n) – десятичные числа с точностью (например, цены: DECIMAL(10,2)).
FLOAT / DOUBLE – числа с плавающей точкой (например, измерения).
BOOLEAN / TINYINT(1) – логическое значение (0 = false, 1 = true).
Используем, когда хотим сохранить количество, цену, логические значения.

2. Текстовые типы
CHAR(n) – текст фиксированной длины (используется реже).
VARCHAR(n) – текст переменной длины (например, имя, e-mail).
TEXT – длинный текст (например, описание продукта).
ENUM – список разрешенных значений (например, статус заказа).
Используем для хранения имен, описаний, текстов, статусов.

3. Типы даты и времени
DATE – только дата (ГГГГ-ММ-ДД).
DATETIME – дата и время (ГГГГ-ММ-ДД ЧЧ:ММ:СС).
TIMESTAMP – дата и время, сохранение с учетом временной зоны.
TIME – только время.
YEAR – только год.
Используем для хранения даты регистрации, логина, времени событий.
